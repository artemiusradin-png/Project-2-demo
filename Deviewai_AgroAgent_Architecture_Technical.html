07 · AI Functioning — How the Engine Stays Honest

Engine: Claude (multimodal) invoked via Vertex AI. Around it we layer constraints so the model output is never user-facing on its own. RAG is necessary but not sufficient; each layer below defends against a distinct failure mode.

┌── ENGINE ─────────────────────────────────────────────────────────────────┐ │ Claude (Vertex AI Model Garden) · multimodal · prompt + image │ └─────────────────────────────────┬─────────────────────────────────────────┘ │ output candidate ▼ ┌── RUNTIME DEFENCES · applied on every request ────────────────────────────┐ │ 1 · RAG over catalog only real SKUs can be cited │ │ 2 · Confidence pre-flight gate refuses below threshold (< 70%) │ │ 3 · Hard regulatory filter approved_in_ua = TRUE (SQL WHERE clause) │ │ 4 · Live Salesforce stock query real availability, never cached │ │ 5 · Deterministic SQL for nums LLM narrates; numbers computed in SQL │ └─────────────────────────────────┬─────────────────────────────────────────┘ ▼ ┌── PER-REQUEST AUDIT ──────────────────────────────────────────────────────┐ │ 6 · Reasoning-chain audit log symptom → diagnosis → rule → SKU + conf │ └─────────────────────────────────┬─────────────────────────────────────────┘ ▼ ┌── HUMAN-IN-THE-LOOP ──────────────────────────────────────────────────────┐ │ 7 · Consultant / supervisor override human approves before C binding │ └─────────────────────────────────┬─────────────────────────────────────────┘ ▼ User-facing answer │ ┌──────────────────────────┴──────────────────────────┐ ▼ ▼ ┌── PRE-LAUNCH VALIDATION ─────────┐ ┌── POST-LAUNCH VALIDATION ─────────┐ │ 8 · Blind agronomist panel │ │ 9 · Field-outcome tracking │ │ ≥ 85% on held-out test set │ │ real disease progression / │ │ (independent of consultants) │ │ yield outcomes, first season │ └──────────────────────────────────┘ └───────────────────────────────────┘
LayerDefends againstWhere
1 · RAG over catalogFabricated products / SKUs that do not exist in the catalogA · pgvector retrieval
2 · Confidence pre-flight gateConfident-wrong diagnoses below trust thresholdA · pipeline gate
3 · Hard regulatory filterRecommending products not approved for use in UkraineSQL WHERE approved_in_ua = TRUE on the catalog query
4 · Live Salesforce stock queryRecommending out-of-stock productsA · live SF REST per request, never cached
5 · Deterministic SQL for numbersHallucinated metrics in vendor reports (margins, conversions, totals)E · data-mart materialised views; LLM only narrates
6 · Reasoning-chain audit logWrong answers being unaccountable / unexplainablePostgreSQL append-only audit table; every prescription logs symptom → diagnosis → rule → SKU + confidence
7 · Consultant / supervisor overrideModel "winning" against a human expert in the fieldC · approval flow before any quote is binding; overrides logged
8 · Blind agronomist panel (pre-launch)The system being subtly wrong on averageEval framework · ≥ 85% gate before production launch
9 · Field-outcome tracking (post-launch)Long-tail drift; seasonal blind spots; v2 retraining inputsContinuous; validates the panel score against reality over the first season
▸ One-liner: the AI never gets the last word — the data does.

08 · Cross-Agent

ConcernDesign
Event flowA emits diagnosis/Rx events → C consumes, emits transaction events → E aggregates A+C+SF. One append-only log, one PostgreSQL instance, no message broker.
SF write isolationOnly C writes (draft-only). A, E read-only. Three scoped permission sets.
Failure isolationNo agent in another's critical path. C down → A still diagnoses; E ELT fails → A, C unaffected.
v1 feedback boundaryA-feedback (retraining / relevance learning) is not automatic in v1. Loop closes via human action (agronomist review → catalog/inventory update). Automatic feedback deferred post-v1.
PhasingA ships v1 (foundation). C, E attach post-v1 as contained increments.
