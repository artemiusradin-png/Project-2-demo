3AI Integration Approach — Flutter ↔ AI Backend

The architecture itself is the competence signal. These are testable, reviewable design facts we commit to from day one.

LayerApproach
Flutter ↔ Backend contractOpenAPI spec auto-generated by FastAPI; typed Dart client via openapi-generator. Pinned, version-controlled, never hand-rolled. Pact contract tests catch schema drift in CI.
Photo pipelineClient-side compression to JPEG ≤800 KB → signed-URL upload to Cloud Storage → backend strips EXIF/GPS (Pillow) → Claude multimodal call. The app never invokes an LLM directly. Photos auto-delete 24 h.
State managementAI response state isolated from agronomic domain state, using the existing app's state pattern (Provider / Riverpod). UI never blocks on inference — loading shell + cached last result while the call resolves.
Cost / latency controlServer-side LLM call cached by deterministic hash (field + crop + week) for repeat lookups. Per-stage latency logged; p99 < 25 s gated in staging.
AuthFirebase Auth on the device; backend validates the JWT on every request. Salesforce access via a scoped service account on the backend — no user-level SF tokens cross the wire.
Failure UXStructured errors with specific Ukrainian messages. Partial result if a downstream (e.g. live stock) fails. No silent wrong answers.

4Offline Logic — the v1 Scope Decision

The brief asked about Flutter-AI integration with offline-logic support where necessary. We've made this a deliberate, documented scope choice for v1.

  • Agro-Vision in v1 uses a server-side multimodal LLM (Claude via Vertex AI). By nature, that call requires connectivity at the moment of diagnosis.
  • A meaningful offline fallback would be a separate workstream — either a small on-device classifier with a narrow crop/problem scope, or a cached rules-and-symptoms decision tree — each with its own UX, cost, and accuracy model.
  • v1 design choice: connectivity-required. Field consultants who have signal get the full diagnosis; those who don't see a clear "connectivity required, photo queued" state — not a silent failure.
  • v2 path designed in: the photo-upload queue, the local result cache, and the result-render path are structured to accept an on-device inference source without re-architecting the app shell. Offline is a roadmap item, not a redesign.
